MYHxCARE Phase 1 Backend Architecture Constitution

MYHxCARE Phase 1 Backend Architecture Constitution

Deployment Target: Nnamdi Azikiwe University Medical Centre, UniZik
Phase: Phase 1
Backend Stack: Go, PostgreSQL, Redis, S3-compatible storage, WebSockets, FCM, Docker
Architecture Style: Clean Architecture, Domain Driven Design, Modular Monolith, Secure by Design, Audit by Design, Privacy by Design, Multi-Tenant Ready
Notification Policy: In-app notifications and Firebase Cloud Messaging only. SMS and paid email services are excluded from Phase 1.
Event Policy: PostgreSQL outbox and background workers for Phase 1. Kafka is deferred until event volume, integrations, or multi-institution deployment justify it.

1. Executive Summary

Business Purpose

MYHxCARE Phase 1 provides a secure, auditable, institution-ready healthcare backend for UniZik Medical Centre. It supports patient records, clinical workflows, wards, pharmacy, laboratory, billing, emergency alerts, collaboration, mobile access, administration, audit logging, and reporting.

The backend must serve approximately 60,000 users/personnel while remaining maintainable enough for a 10-week delivery timeline.

Technical Design

The system is a Go modular monolith deployed on DigitalOcean Droplets using Docker. PostgreSQL is the source of truth. Redis supports caching, rate limiting, distributed locks, session state, and notification state. Object files are stored in S3-compatible storage, preferably DigitalOcean Spaces for Phase 1. WebSockets support real-time collaboration, read receipts, presence, in-app notifications, and emergency alerts. FCM supports push notification delivery to mobile devices.

Implementation Guidance

Build one deployable backend application with internal domain modules. Each domain owns its business rules, repository interfaces, service/use-case layer, HTTP handlers, validation, and audit emission. Shared infrastructure is kept behind interfaces.

Risks

  • Overengineering could delay the 10-week go-live.
  • Underengineering clinical immutability, audit, and permissions could create compliance and safety risks.
  • Running too many services separately increases cost and operational load.

Mitigation Strategies

  • Use a modular monolith instead of microservices.
  • Use PostgreSQL outbox instead of Kafka for Phase 1.
  • Make every critical action auditable from day one.
  • Keep tenant columns in all tables even though only UniZik is deployed initially.

Best Practices

  • Prefer boring, reliable technology.
  • Use strict boundaries between domain, application, infrastructure, and transport layers.
  • Never allow HTTP handlers to access database code directly.
  • Treat clinical records, audit records, collaboration messages, and financial records as append-only or versioned.

2. Core Architecture Decisions

Business Purpose

The architecture must let the team ship Phase 1 quickly without blocking future expansion to other institutions.

Technical Design

Area Decision Justification
Backend language Go latest stable Strong performance, low memory use, simple deployment, excellent concurrency
HTTP framework Gin Mature ecosystem, fast routing, common team familiarity, enough middleware support
Database PostgreSQL Strong consistency, relational integrity, JSONB support, transactions, reporting
Cache/locks Redis Fast cache, rate limits, distributed locks, ephemeral state
Events PostgreSQL outbox Reliable, simple, transactional, cheaper than Kafka for Phase 1
Future events Kafka Add later for multi-campus/multi-tenant event streaming
Object storage DigitalOcean Spaces or S3-compatible storage Cloud-portable object storage API
Notifications In-app + FCM Eliminates SMS/email recurring cost while supporting mobile alerts
Deployment Docker on DigitalOcean Droplets Predictable cost, simpler operations
Architecture Modular monolith Faster delivery, simpler debugging, fewer distributed failure modes

Implementation Guidance

Use Gin for routing, request binding, middleware, and response handling. Keep Gin-specific code in internal/interfaces/http. Domain and application layers must not import Gin.

Risks

  • Gin can encourage handler-heavy code if boundaries are not enforced.
  • PostgreSQL outbox can become a bottleneck if not indexed and drained properly.
  • DigitalOcean self-hosting requires operational discipline.

Mitigation Strategies

  • Enforce package dependency rules in code review.
  • Use outbox partitioning or status/date indexes.
  • Configure backups, metrics, log retention, and restore drills before go-live.

Best Practices

  • Use context.Context everywhere.
  • Use transactions explicitly for state-changing workflows.
  • Use UUID primary keys only.
  • Use structured logs and correlation IDs on every request.

3. Modular Monolith Rationale

Business Purpose

Phase 1 must deliver hospital functionality rapidly and reliably. The team should not spend the first release solving microservice networking, deployment, distributed transactions, and observability complexity.

Technical Design

The backend is one application containing domain modules:

  • Identity and Access
  • Admissions and Records
  • Clinicals
  • Clinical Collaboration
  • Duty Roster
  • Ward Management
  • Pharmacy
  • Multi-Campus Pharmacy Extension
  • Laboratory
  • Billing and Finance
  • Department Revenue Accounting
  • NHIA/NHIS
  • Emergency
  • Student Mobile App Backend
  • HMS Administration
  • Audit
  • Notifications
  • Reporting

Each module has its own application services, domain models, repository interfaces, infrastructure repository implementation, HTTP handlers, DTOs, and tests.

Implementation Guidance

Do not create separate services in Phase 1. Instead, create package boundaries strict enough that future extraction is possible.

Future extraction candidates:

Future Service Extraction Trigger
Notifications Very high real-time or FCM volume
Pharmacy Multi-campus inventory complexity grows
Laboratory External lab equipment integrations increase
Billing External payment/insurance integrations increase
Reporting Heavy analytics affects transactional database
Audit Audit volume requires separate write-optimized store

Risks

  • Poor module boundaries can turn the system into a tangled monolith.
  • Premature service extraction could delay delivery.

Mitigation Strategies

  • Use domain interfaces, not cross-module database calls.
  • Enforce package-level dependency checks in code review.
  • Store outbox events with clear domain event names.

Best Practices

  • One process, many internal modules.
  • Shared kernel must be tiny.
  • Cross-domain communication should happen through application services or domain events, not direct table writes.

4. Multi-Tenancy Foundation

Business Purpose

Phase 1 serves UniZik only, but the backend must support future universities, polytechnics, colleges, and teaching hospitals without redesign.

Technical Design

Every tenant-owned table must contain:

  • tenant_id uuid not null
  • id uuid not null
  • created_at timestamptz not null
  • updated_at timestamptz not null
  • deleted_at timestamptz null
  • created_by uuid null
  • updated_by uuid null
  • version integer not null default 1

Phase 1 uses one preconfigured tenant row for UniZik. No tenant onboarding UI or SaaS administration is included.

Tenant Context Strategy

The authenticated access token contains tenant_id. The request middleware extracts and validates it, then stores it in request context.

Required context values:

Context Key Source
tenant_id JWT claim
actor_id JWT subject
role_ids JWT/session claims
request_id Gateway/API middleware
correlation_id Client header or generated

Tenant Resolution Strategy

For Phase 1:

  1. Resolve tenant from JWT.
  2. Reject requests without tenant_id except login, refresh, health checks, and public metadata endpoints.
  3. Validate tenant status is active.

Future strategies may include subdomain, institution code, or enterprise SSO mapping, but these are not implemented in Phase 1.

Tenant Isolation Strategy

  • All queries must include tenant_id.
  • Composite indexes must usually start with tenant_id.
  • Foreign keys should include tenant consistency through application validation and, where practical, composite references.
  • Background workers must process events using the tenant_id stored in the outbox row.

Risks

  • Missing tenant_id filters can leak data between institutions later.
  • Developers may forget tenant scope in raw SQL.

Mitigation Strategies

  • Repository methods must require tenantID uuid.UUID.
  • Use static analysis/code review checklist for tenant filters.
  • Add integration tests proving tenant isolation.

Best Practices

  • Never infer tenant from request body.
  • Never allow client-supplied tenant IDs to override JWT tenant context.
  • Never create tenant onboarding functionality in Phase 1.

  • Inconsistent route names slow frontend/mobile development.

  • Do not expose internal IDs from third-party systems as primary identifiers.

8. Database Governance

Business Purpose

PostgreSQL stores clinical, operational, financial, audit, and administrative data with strong consistency.

Technical Design

Use UUID primary keys. Never use auto-increment IDs.

Every table includes:

id uuid primary key,
tenant_id uuid not null,
created_at timestamptz not null default now(),
updated_at timestamptz not null default now(),
deleted_at timestamptz null,
created_by uuid null,
updated_by uuid null,
version integer not null default 1

Standards

Item Standard
Table names plural snake_case, e.g. patients, ward_beds
Columns snake_case
Primary keys id uuid
Foreign keys {resource}_id uuid
Timestamps timestamptz
Soft delete deleted_at
Optimistic locking version
Money numeric(12,2) plus currency column if needed
Clinical text structured columns plus JSONB only where flexible content is required

Indexing Strategy

  • Add indexes on all foreign keys.
  • Add composite indexes beginning with tenant_id.
  • Add partial indexes for active rows: where deleted_at is null.
  • Use unique indexes with tenant scope.

Example:

create unique index ux_beds_active_code
on ward_beds (tenant_id, ward_id, bed_code)
where deleted_at is null;

Migration Strategy

  • Use a migration tool such as Goose or golang-migrate.
  • All migrations must be reversible unless destructive archival is explicitly approved.
  • Schema migrations must run in CI against an empty database and an upgraded existing database.

Backup Strategy

  • Daily DigitalOcean Droplet backups.
  • Nightly logical PostgreSQL dumps to Spaces.
  • Weekly full backup restore test in staging.
  • Retain daily backups for 14 days, weekly for 8 weeks, monthly for 12 months.

Archival Strategy

  • Clinical, audit, collaboration, billing, and inventory records are not physically deleted.
  • Use soft delete only for administrative lookup data when allowed.
  • Archive old reporting snapshots to object storage when database growth affects performance.

Risks

  • Missing constraints can corrupt clinical and financial data.
  • Heavy reports can slow transactional workloads.

Mitigation Strategies

  • Use transactions for admissions, bed allocation, dispensing, billing, and result verification.
  • Use read replica for reports.
  • Add materialized views for dashboards where needed.

Best Practices

  • Do not use application-only uniqueness when database uniqueness is required.
  • Use SELECT ... FOR UPDATE for critical allocation and stock operations.
  • Store audit and outbox entries in the same transaction as the business change.

9. Redis Governance

Business Purpose

Redis supports fast ephemeral operations but is not the system of record.

Technical Design

Redis is used for:

  • Session revocation state
  • Refresh token family state
  • Rate limiting counters
  • Distributed locks
  • Short-lived cache
  • WebSocket presence
  • Notification delivery state

Key Naming Conventions

myhxcare:{env}:{tenant_id}:auth:session:{session_id}
myhxcare:{env}:{tenant_id}:auth:revoked:{token_id}
myhxcare:{env}:{tenant_id}:rate:{scope}:{identifier}
myhxcare:{env}:{tenant_id}:lock:{resource}:{id}
myhxcare:{env}:{tenant_id}:presence:user:{user_id}
myhxcare:{env}:{tenant_id}:notification:unread:{user_id}

Implementation Guidance

  • Locks must have TTL.
  • Cache keys must have explicit TTL.
  • Redis failure must degrade gracefully for non-critical caches.
  • Redis failure must fail closed for authentication/session revocation checks if security depends on it.

Risks

  • Redis memory exhaustion can affect sessions and locks.
  • Incorrect lock handling can cause deadlocks.

Mitigation Strategies

  • Set max memory policy.
  • Monitor memory and eviction.
  • Use lock tokens and safe unlock scripts.

Best Practices

  • Never store permanent clinical records only in Redis.
  • Never store plaintext secrets.
  • Keep Redis access behind an interface.

10. Security Governance

Business Purpose

MYHxCARE handles sensitive health, identity, financial, and operational data. Security must be built into every layer.

Technical Design

Area Standard
Authentication JWT access token + refresh token
Access token TTL 10-15 minutes
Refresh token TTL 7-30 days, configurable
Password hashing Argon2id preferred, bcrypt acceptable
Authorization RBAC with permissions
Session revocation Redis + persisted session table
MFA Schema and flow readiness, optional in Phase 1
Transport TLS everywhere
Secrets Environment variables or secret manager, never committed
Audit All critical actions logged
Rate limiting Per IP, user, route, and auth action

Authorization Model

Use permissions, not hardcoded role names. Roles are collections of permissions.

Example permissions:

patients.read
patients.write
clinical_notes.create
clinical_notes.read_assigned
prescriptions.create
pharmacy.dispense
ward_beds.allocate
billing.refund
audit.read
admin.staff.manage

Implementation Guidance

  • Middleware authenticates token and loads session.
  • Application service checks permission and resource-level access.
  • Clinical access must be purpose-bound: a user must be assigned, on duty, referred, or explicitly authorized.
  • Administrators cannot access clinical records unless clinically authorized.

Risks

  • Role sprawl can create excessive privileges.
  • Admin bypass of clinical privacy could violate trust and policy.
  • Weak token revocation can leave stolen sessions active.

Mitigation Strategies

  • Use least privilege.
  • Review roles before go-live.
  • Store refresh token hashes, not raw tokens.
  • Rotate refresh token families on use.

Best Practices

  • Validate input at DTO and domain layers.
  • Use parameterized SQL only.
  • Set secure HTTP headers.
  • Use CSRF protection for cookie-based browser flows.
  • Use CORS allowlist.
  • Apply OWASP Top 10 controls.

Executive Summary

Identity controls login, authorization, sessions, device registration, token refresh, password reset, MFA readiness, and device revocation.

Only verified users should access MYHxCARE, and each user should see only the workflows and records they are authorized to access.

  • User
  • Role
  • Session
  • Device
  • PasswordResetRequest

  • Role

Value objects:

  • PhoneNumber, optional, not used for SMS delivery
  • PasswordHash
  • DeviceFingerprint
  • JWTClaims
  • PermissionCode

Repositories:

  • UserRepository
  • RoleRepository
  • SessionRepository
  • DeviceRepository
  • PasswordResetRepository

Services:

  • AuthService
  • AuthorizationService
  • SessionService
  • PasswordResetService
  • DeviceService
  • MFAReadinessService

Core endpoints:

POST   /api/v1/auth/login
POST   /api/v1/auth/refresh

POST   /api/v1/auth/password-resets/confirm

GET    /api/v1/roles

Implementation Guidance

  • Store password hashes with Argon2id or bcrypt.
  • Store refresh token hashes only.
  • Rotate refresh tokens on every refresh.
  • Persist sessions in PostgreSQL and store hot revocation state in Redis.
  • Device registration stores device ID, platform, FCM token hash/reference, last seen timestamp, and revocation status.
  • MFA-ready schema includes mfa_enabled, mfa_method, mfa_secret_encrypted, recovery_codes_hash.

Validation Rules

  • Password minimum length: 12 characters for staff/admin, configurable.
  • Lock account temporarily after repeated failed attempts.
  • Refresh token must belong to a valid token family and session.
  • Device registration requires authenticated user.

Security Rules

  • Login is rate-limited by IP and username.
  • Admin role changes require elevated permission.
  • Staff cannot assign permissions they do not possess unless super-admin equivalent policy allows it.

Audit Requirements

Audit login success/failure, logout, refresh token rotation failure, password reset request, password reset completion, role change, permission change, device registration, and device revocation.

Reporting Requirements

  • Active users
  • Failed login trends
  • Session counts
  • Privileged users
  • Device registration counts

Error Handling

Use generic login error: INVALID_CREDENTIALS. Do not reveal whether username or password failed.

Performance Considerations

Cache permission maps briefly in Redis. Keep login P95 under 150ms excluding password hashing cost; hashing cost must remain secure and acceptable.

Risks

  • Token theft
  • Excess privileges
  • Weak passwords

Mitigation Strategies

  • Refresh token rotation
  • Session revocation
  • Least privilege roles
  • Rate limits and lockouts

Best Practices

  • Never log tokens or passwords.
  • Use short-lived access tokens.
  • Use resource-level authorization for clinical data.

16. Admissions and Records Domain

Executive Summary

Admissions and Records manages student/staff registration, dependents, master patient index, demographics, folders, and patient identity resolution.

Business Purpose

  • Emergency write path persists alert, audit, and outbox event in one transaction.
  • WebSocket push goes to active responders.
  • FCM push goes to registered devices.
  • Duty roster resolves active responders.
  • If no responder acknowledges within configured interval, escalate.

Validation Rules

  • Emergency type required.
  • GPS data must include latitude and longitude when available.
  • Acknowledgement requires active alert.

Security Rules

  • Authenticated students/mobile users may raise emergency reports if policy allows.
  • Staff acknowledgement requires responder authorization.

Audit Requirements

Audit alert creation, delivery attempt, acknowledgement, escalation, and resolution.

Reporting Requirements

  • Response time
  • Acknowledgement time
  • Emergency volume
  • Escalation counts

Risks

  • Slow alert fanout
  • FCM delay outside system control
  • Duty roster gaps

Mitigation Strategies

  • Prioritized outbox queue
  • WebSocket first for active dashboards
  • Duty coverage dashboard
  • Emergency route performance testing

Best Practices

  • Keep emergency payload small.
  • Always persist before notification.
  • Alert response target is measured from API acceptance to local in-app/WebSocket delivery attempt.

32. Notification Architecture

Executive Summary

Phase 1 uses in-app notifications and FCM push only. SMS and paid transactional email are excluded.

Business Purpose

Reduce recurring cost while still providing operational alerts to web and mobile users.

Technical Design

Notification channels:

  • In-app notification stored in PostgreSQL
  • WebSocket real-time delivery
  • FCM push notification to registered devices

Implementation Guidance

  • Create notification row in PostgreSQL.
  • Create outbox event in same transaction.
  • Worker reads outbox, sends WebSocket event, sends FCM push if device exists.
  • FCM payload must be minimal.

Risks

  • FCM delivery is not guaranteed or instant.
  • Users may disable push notifications.

Mitigation Strategies

  • In-app notification centre is source of truth.
  • Emergency dashboards use WebSocket and visible in-app state.
  • Retry failed FCM attempts with backoff.

Best Practices

  • Store delivery attempts.
  • Allow notification preferences where clinically safe.

36. Final Implementation Mandate

MYHxCARE Phase 1 must be implemented as a secure, auditable, multi-tenant-ready Go modular monolith. The deployment remains single-tenant for UniZik. PostgreSQL is the source of truth. Kafka is deferred; PostgreSQL outbox is the Phase 1 event backbone.
